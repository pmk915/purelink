# PureLink M5 Upgrade Plan: Retrieval Trace

## Goal

M5 makes retrieval observable. PureLink should record enough structured detail to debug why a RAG answer used specific evidence without changing answer behavior.

## Scope

- Add `retrieval_traces` and `retrieval_trace_items`.
- Create trace records from `retrieval_service.retrieve()`.
- Record query, mode, top_k, provider metadata, candidate counts, final evidence counts, and reranker usage.
- Record candidate-level document/chunk/citation references, scores, ranks, selected status, and filtered reasons.
- Keep trace text previews truncated.
- Keep trace write failures from breaking normal retrieval.

## Non-goals

- No GraphRAG, entity extraction, DocumentBlock schema, RAG evaluation harness, agent runtime, LangChain, or LangGraph.
- No frontend trace visualization in M5.
- No full prompt storage.
- No retrieval ranking changes.

## Design

`retrieval_traces` is the trace header:

- user and knowledge base context
- query and retrieval mode
- top_k and candidate/evidence counts
- embedding provider/model
- reranker provider/model and usage
- completion timing and metadata

`retrieval_trace_items` records candidate-level details:

- document/chunk/citation unit references
- truncated text preview
- vector, keyword, rerank, and final scores
- initial, rerank, and final ranks
- selected_for_context
- filtered_reason
- index compatibility metadata when available

Trace records are generated by the retrieval layer. They are for debugging and later evaluation, not for answer generation.

## Compatibility

Ask and retrieve responses remain compatible. `RetrievalResult.trace_id` is populated internally when tracing is enabled, but no public trace API is exposed in M5.

## Future Work

M5 prepares for:

- RAG evaluation over trace data
- GraphRAG candidate comparison
- frontend trace inspection
- agent debugging
- safer model/index migration analysis
