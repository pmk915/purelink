# PureLink 开发计划

## 项目定位

PureLink 是一个面向团队内部文档管理的 AI 知识库问答与任务助手平台。

当前技术路线：

- 后端：FastAPI
- 数据库：PostgreSQL
- 缓存：Redis
- AI：Embedding + 向量库 + RAG
- 部署：Docker

## 当前状态

已完成里程碑：

- `M1`：API 结构规范化与应用入口整理
- `M2`：数据库底座与核心业务实体
- `M2.5`：数据库迁移与本地验证链路
- `M3`：最小认证系统
- `M4`：个人知识库 CRUD
- `M4.5`：团队域模型
- `M4.6`：团队创建、邀请码、入队与成员查看

当前已具备能力：

- 用户注册、登录、当前用户查询
- JWT 认证与当前用户依赖
- 个人知识库 CRUD 与归属隔离
- 团队创建、团队列表、团队详情
- 管理员生成邀请码、查看邀请码
- 用户通过邀请码加入团队
- 团队成员列表
- PostgreSQL + SQLAlchemy + Alembic 迁移基础

## 当前阶段目标

当前阶段的目标不是直接进入 RAG，而是把“多租户协作底座”补稳，为后续团队知识库、文档审核、处理流水线和检索访问控制做准备。

当前阶段优先级：

1. 团队知识库
2. 文档上传与审核流
3. 处理状态与异步边界
4. 检索权限控制

## 路线图

### M4.7 团队知识库

**Goal**

让团队拥有独立于个人知识库的知识库视图和管理入口。

**Scope**

- 创建团队知识库
- 列出我可访问的团队知识库
- 团队管理员更新/删除团队知识库
- 清晰区分个人知识库与团队知识库接口语义

**Out of Scope**

- 文档上传
- 文档审核
- 检索与问答

**Acceptance Criteria**

- 团队 `admin` 可以创建团队知识库
- 团队成员可以查看自己可访问的团队知识库
- 非团队成员无法访问团队知识库
- 现有个人知识库 CRUD 不回归

### M5 文档上传与审核流

**Goal**

建立个人知识库直通上传、团队知识库审核上传的基础流程。

**Scope**

- 上传文档到个人知识库
- 提交文档到团队知识库
- 团队成员上传的文档默认进入 `pending_review`
- 团队管理员批准或拒绝文档

**Out of Scope**

- 文档解析
- 索引构建
- 检索问答

**Acceptance Criteria**

- 个人知识库文档默认 `review_status=not_required`
- 团队知识库成员上传默认 `review_status=pending_review`
- 只有团队 `admin` 可以做审核决定
- 非成员不能向团队知识库上传

### M6 文档解析与异步处理

**Goal**

把文档从“已上传”推进到“可被索引”的处理中间态。

**Scope**

- 文件解析边界
- chunking 预留
- `processing_status` 更新
- 重试与失败状态
- 为未来 Go worker 预留清晰集成边界

**Out of Scope**

- 真正的向量检索
- Q&A 接口

**Acceptance Criteria**

- 文档处理状态变化有清晰定义
- 失败与重试场景有最小建模
- 处理逻辑不耦合具体 RAG 框架

### M7 索引与检索

**Goal**

接入 Embedding、向量索引和带权限过滤的检索能力。

**Scope**

- Embedding 生成
- 向量索引写入
- 检索访问控制
- 仅允许 `approved` 的团队文档进入检索范围

**Out of Scope**

- 问答编排优化
- 引文渲染打磨

**Acceptance Criteria**

- 个人知识库只能被所有者检索
- 团队知识库只能被成员检索
- `pending_review/rejected` 团队文档不会进入检索范围

### M8 问答能力

**Goal**

提供基于个人知识库和团队知识库的问答能力。

**Scope**

- 个人知识库问答
- 团队知识库问答
- 引文片段
- 访问控制

**Out of Scope**

- 复杂多轮工作流
- 自动任务执行

**Acceptance Criteria**

- 个人问答只允许所有者使用
- 团队问答只允许团队成员使用
- 响应能返回最小引文信息

### M9 Go 高并发组件

**Goal**

只在确实需要高并发和 worker 模式时引入 Go 组件。

**Scope**

- 文档异步处理 worker
- 索引 worker
- 队列消费者
- 实时进度/通知服务
- 高并发文件处理

**Out of Scope**

- 替换主后端

**Acceptance Criteria**

- 主应用仍保持 FastAPI
- Go 只承担高并发、队列、worker 类职责
- 两端边界通过清晰接口或任务协议连接

## 当前约束

- 保持现有个人知识库行为继续可用
- 不提前引入厚重的 repository 抽象
- 服务层保持薄且业务导向
- 权限判断以所有权和成员关系为中心
- `owner_id`、`team_id`、审核结果等特权字段不能由前端直接决定
- 涉及结构性迁移时，至少保证 `upgrade head` 可验证
- 后续关键迁移应逐步补 `upgrade -> downgrade -> upgrade` 验证

## 下一步建议

建议优先进入 `M4.7 团队知识库`，不要直接跳到 RAG。

原因：

- 团队知识库是文档上传、审核、检索权限的直接宿主
- 没有团队知识库层，后续审核流和检索权限会变得不清晰
- 当前模型和团队成员/邀请码已经具备承接条件
