# AGENTS.md

## 目的

本文件用于约束我在 PureLink 仓库中的工作方式，确保后续实现始终围绕当前产品阶段推进，而不是脱离路线图做过早扩写。

## 当前阶段定位

- 项目名称：PureLink
- 当前阶段：认证、个人知识库、团队协作基础能力
- 已完成：用户认证、个人知识库 CRUD、团队域模型、团队创建与邀请码入队
- 下一重点：团队知识库、文档上传与审核流

## 工作原则

- 先看现状，再改代码，不基于猜测扩写
- 先补计划和边界，再进入实现
- 保持 diff 小而清晰，优先做局部、可审查的改动
- 单次改动优先解决一个清晰业务目标
- 若文档与代码状态不一致，先同步文档

## 代码与目录约束

- `app/main.py` 只承担应用入口
- `app/api` 负责路由和权限入口，不承载复杂业务编排
- `app/services` 负责薄服务层，按业务场景组织
- `app/schemas` 负责请求/响应模型
- `app/models` 负责 SQLAlchemy ORM 模型与关系
- `app/core` 负责配置、日志、安全、异常处理等基础设施
- `app/db` 负责 Base、engine、session、迁移接线
- `tests` 只补当前阶段直接相关的验证，不堆积未来场景
- `docs` 存放设计说明、领域文档和阶段性技术约束

## 实现边界

- 不提前引入复杂权限框架；优先使用清晰的 ownership/membership/admin 校验
- 不提前引入厚重的 repository 层；优先保持 service 薄且直接
- 不为了“未来可能需求”一次性创建大量空模块
- 不提前实现 Embedding、向量检索、RAG 编排，除非当前里程碑明确要求
- 不让前端直接决定 `owner_id`、`team_id`、审核结果等特权字段

## 接口约束

- 所有业务接口统一挂载到 `/api/v1`
- 需要登录的接口必须复用现有认证依赖
- 个人资源优先按 `owner_id` 做隔离
- 团队资源优先按有效成员关系做隔离
- 管理员操作优先按 `TeamMember.role == admin` 校验

## 数据与迁移约束

- 模型字段和枚举命名优先清晰、稳定、面向业务
- 涉及结构变更时优先手写 Alembic 迁移，而不是盲目依赖自动生成
- SQLite 迁移验证有价值，但不能替代 PostgreSQL 验证
- 关键迁移至少要验证一次 `upgrade head`
- 后续复杂迁移应逐步补 `upgrade -> downgrade -> upgrade`
- 若 downgrade 未验证，需要在结果说明里明确标出

## 文档同步要求

- 修改项目结构时同步更新 `README.md`
- 修改阶段路线时同步更新 `PLAN.md`
- 修改领域模型边界时同步更新 `docs/`
- 若工作内容已经突破本文件约束，需要先更新本文件再继续实现

## 测试与验证要求

- 每次完成功能后至少跑与该功能直接相关的测试
- 若改动影响权限或归属逻辑，需要覆盖成功路径和失败路径
- 若改动影响数据库结构，需要说明迁移验证范围
- 若无法完成验证，要明确说明缺失条件和阻塞点

## 提交风格

- 优先小步、可验证的改动
- 结果说明以“做了什么 + 为什么 + 如何验证”为主
- 不把文档写成和代码脱节的理想设计稿
